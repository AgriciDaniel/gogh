Prompt Layer vs Toolchain explains when taste prompting is enough and when persistent contracts plus detectors justify their setup and context cost.

## What it is
Taste Skill and Anthropic frontend-design are prompt-layer guidance in the stack map.
Impeccable is the main toolchain enforcement layer in the stack map.
Taste prompting can set direction, bans, and review expectations without installing hooks.
Impeccable init writes PRODUCT.md and offers DESIGN.md, per claim-pack-impeccable I009.
Impeccable detect provides 45 deterministic detector rules as of 2026-07-07.
The Impeccable March 2026 token-cost anecdote reports 8,000 to 15,000 tokens per session and 10 to 15 percent overhead.
The token-cost figures describe the v1.x seven-reference-file design, not the current v3.9.1 compiled skill.
The claim pack found a second independent practitioner source for qualitative token-overhead concurrence.
The exact token numbers remain single-source.
The current source ledger records Impeccable skill-v3.9.1 and cli-v3.2.0 as of 2026-07-07.
The comparison therefore treats cost claims as practitioner caveats, not current measured benchmarks.
The decision is economic: persistent enforcement pays off when repeated review cost exceeds setup cost.

## How it works
Use prompt-layer guidance for a one-off landing page with no ongoing design system.
Use prompt-layer guidance when the agent needs taste direction before any code exists.
Use prompt-layer guidance when the project is exploratory and installing hooks would be premature.
Use toolchain contracts when the project has repeated screens or multiple sessions.
Use toolchain contracts when brand, audience, anti-references, colors, type, and components must persist.
Use deterministic detectors when common anti-patterns keep returning.
Use hooks when edit-time feedback is more valuable than after-the-fact review.
Use CI detect when merge gates need repeatability without an LLM.
Use prompt-layer plus Vercel audit when the work is one-off but accessibility and UX findings still matter.
Use prompt-layer plus MIFB when feel and polish matter but persistent contracts do not.
Use UI UX Pro Max retrieval when the problem is finding candidates rather than enforcing policy.
Use the product boundary rule: Impeccable detector results remain upstream Impeccable results.
The local stack advisor always includes Impeccable in recommended stacks, but an operator can still decide how hard to enforce it.
The comparison does not claim a current v3.9.1 token benchmark.

> [!gap]
> The exact Impeccable token-cost numbers are practitioner and v1.x-specific. A current v3.9.1 measurement remains open.

## Best practice
- Start with prompt-layer direction when the design question is still aesthetic or conceptual EVIDENCE-BASED
- Move to Impeccable contracts when the project needs persistent audience, brand, voice, and component context EVIDENCE-BASED
- Use deterministic detect when a repeatable no-LLM gate is needed EVIDENCE-BASED
- Treat March 2026 token-cost numbers as v1.x practitioner evidence only PRACTITIONER
- Do not cite the v1.x token overhead as a v3.9.1 measurement CONTESTED
- Use hooks when direct UI edits should surface detector findings during work EVIDENCE-BASED
- Use prompt guidance alone when setup cost is higher than project reuse value PRACTITIONER
- Record contract waivers for legitimate brand exceptions PRACTITIONER
- Keep Vercel audit as a review layer even when prompt guidance is enough EVIDENCE-BASED
- Keep UI UX Pro Max as retrieval, not a substitute for enforcement EVIDENCE-BASED

## Pitfalls
Installing a toolchain for a tiny one-off can add process without much value.
Relying only on prompts for a multi-session product can lose design context.
Treating PRODUCT.md and DESIGN.md as optional after Impeccable init can break command quality.
Treating detector output as a design critic can miss subjective aesthetic direction.
Treating a prompt ban as enforcement can lead to repeated drift.
Using old v1.x token numbers as current cost can mislead operators.
Ignoring the hook trust step in Codex can make Impeccable hooks appear broken.
Ignoring existing brand tokens can make both prompt and toolchain layers fight the product.
Treating UI UX Pro Max retrieval as proof of implementation quality can overstate it.
Treating Vercel audit as a generator can misapply its scope.
